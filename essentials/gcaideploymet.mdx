---
title: "Deploying a Streamlit ML App on Google Cloud with Kubernetes and Grafana"
---

If you have built a Streamlit-based Machine Learning or AI application and are wondering how to make it accessible on the internet, this guide walks through a complete cloud-native deployment workflow. By the end, you will deploy a Streamlit app to a Google Cloud VM, run it inside a Minikube Kubernetes cluster using Docker, expose it to the public internet, and optionally monitor it with Grafana Cloud.

## Architecture overview

Before writing any commands, it helps to understand the high-level architecture of the system you are about to deploy.

![Gcp K8s Architecture](/images/gcp_k8s_architecture.png)

The main components are:

<Steps>
  <Step title="Google Cloud VM instance">
    An Ubuntu-based virtual machine provisioned on Google Cloud Compute Engine. This VM acts as the infrastructure layer. It hosts Docker, Minikube, and the Kubernetes workloads that run the Streamlit application.
  </Step>
  <Step title="Minikube Kubernetes cluster">
    A single-node Kubernetes cluster running on the VM that manages pods, services, and deployments for the application.
  </Step>
  <Step title="Docker">
    A containerization platform that bundles the application source code, Python dependencies, and runtime configuration into a portable container image that runs consistently across environments.
  </Step>
  <Step title="Streamlit app">
    A Python web application served on port 8501 inside a Kubernetes pod.
  </Step>
  <Step title="Grafana Cloud">
    A managed observability platform for monitoring Kubernetes resources in real time. Monitored resources include nodes, pods, deployments, CPU usage, memory utilization, and overall cluster health.
  </Step>
</Steps>

## Step 1: Prepare the repository

Before using Google Cloud, verify that the code is organized and pushed to GitHub.

The GitHub repository should include:

- Application code, for example`app.py`, `src/`.
- A `requirements.txt`file listing all Python dependencies with compatible versions.
- A `Dockerfile`describing how to build the container image for the application.
- A Kubernetes manifest file for the deployment and service, for example`llmops-k8s.yaml`.

Keeping these assets version-controlled makes it easy to reproduce deployments and roll back safely.

## Step 2: Create a Google Cloud VM

Create the virtual machine that will host the Kubernetes cluster and application.

1. Sign in to the Google Cloud Console.
2. Click **Compute Engine \> VM instances**, and then click **Create instance**.
3. Set a descriptive name, for example ,`streamlit-app-server`.
4. Under **Machine configuration**, select an E2 machine type.\
   If you are running ML workloads, ensure that the machine type has at least 16 GB of RAM, for example, `e2-standard-4`.
5. Under **Boot disk**, click **Change**, select Ubuntu 24.04 LTS, and increase the disk size to approximately 256 GB to accommodate Docker images and model files.
6. In the **Firewall** section, enable **Allow HTTP traffic** and **Allow HTTPS traffic**.
7. Click **Create** and wait until the VM is ready.
8. Click the **SSH** button next to the instance to open a terminal in the browser.

<Info>
  #### What happens during VM provisioning?

  At this stage, Google Cloud provisions a dedicated Ubuntu server for the deployment. This server hosts the following components:

  - Docker Engine

  - Minikube Kubernetes cluster

  - `kubectl` command-line tool

  - Streamlit application containers

  The VM supports the entire deployment architecture.
</Info>

## Step 3: **Install and configure Docker**

Docker is the containerization platform that packages the Streamlit application, its dependencies, and runtime environment into a portable image that can be deployed consistently across environments.

Docker is also a prerequisite for Minikube, because Minikube uses Docker as its container runtime when creating the Kubernetes cluster.

### **Install Docker**

Complete the following substeps to install Docker Engine.

#### **Install Docker dependencies**

Update the package repository and install the utilities required to download Docker packages securely:

```shell
sudo apt-get update
sudo apt-get install -y ca-certificates curl
```

#### **Add the official Docker GPG key**

Ubuntu verifies package authenticity using GPG keys. The official Docker GPG key ensures that the packages being installed come from a trusted source:

```shell
sudo install -m 0755 -d /etc/apt/keyrings
sudo curl -fsSL https://download.docker.com/linux/ubuntu/gpg \
  -o /etc/apt/keyrings/docker.asc
sudo chmod a+r /etc/apt/keyrings/docker.asc
```

#### **Add the Docker package repository**

Add the official Docker package repository to the system so that Ubuntu can download the latest Docker packages directly from the Docker-maintained repository:

```shell
echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.asc] \
  https://download.docker.com/linux/ubuntu \
  $(. /etc/os-release && echo "$VERSION_CODENAME") stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

sudo apt-get update
```

#### **Install Docker Engine**

Install Docker Engine and its associated plugins:

```shell
sudo apt-get install -y docker-ce docker-ce-cli \
  containerd.io docker-buildx-plugin docker-compose-plugin
```

The installation includes the following components:

- Docker Engine
- Docker CLI
- Container runtime (containerd)
- Buildx plugin
- Docker Compose plugin

#### **Allow Docker usage without** `sudo`

By default, Docker commands require elevated privileges. Add the current user to the Docker group to run Docker commands without using `sudo`:

```shell
sudo usermod -aG docker $USER
```

> **Note:** Run `newgrp docker`, or log out and log in again, for the group membership change to take effect.

#### **Enable Docker on system startup**

Configure Docker services to start automatically when the VM reboots:

```shell
sudo systemctl enable docker.service
sudo systemctl enable containerd.service
```

#### **Verify the installation**

To verify that Docker is installed correctly, run:

```text
docker run hello-world
```

<Note>
  **Note:** If Docker is run without `sudo`, a permission-denied error might occur. This happens because the current user has not yet been added to the Docker user group. The Docker post-installation steps resolve this issue by granting the appropriate permissions.
</Note>

After the configuration is complete, the command displays:

```text
Hello from Docker!
```

This verifies that the Docker daemon is running and can pull and run containers.

#### **Validate the Docker configuration**

To confirm that Docker is operating correctly, run:

```shell
systemctl status docker
docker ps
docker ps -a
```

Expected results:

- The Docker service status shows `active (running)`.
- `docker ps` returns no running containers.
- `docker ps -a` displays the previously run `hello-world` container in an exited state.

<Info>
  #### **Docker in the deployment architecture**

  At this stage, the deployment architecture consists of the following layers:

  - **Google Cloud VM** — the infrastructure layer
  - **Docker Engine** — installed and running on the VM
  - **Container runtime** — ready to build images and run containers

  The VM can now build Docker images and run containers. In the next step, Minikube uses Docker to create a Kubernetes cluster that hosts the Streamlit application.
</Info>

## Step 4: **Install Minikube and create a Kubernetes cluster**

Minikube is a single-node Kubernetes distribution that creates a Kubernetes cluster inside the Google Cloud VM. It uses Docker as its container runtime, making Docker a prerequisite for this step.

### **Kubernetes orchestration**

Rather than running containers directly on the VM, Kubernetes provides an orchestration layer that manages the following resources:

- pods
- deployments
- services
- secrets
- namespaces

Kubernetes enables applications to be deployed, updated, and monitored through a consistent, declarative management model.

### **Install Minikube**

Download and install the Minikube binary:

```shell
# Download Minikube
curl -LO https://storage.googleapis.com/minikube/releases/latest/minikube-linux-amd64

# Install Minikube
sudo install minikube-linux-amd64 /usr/local/bin/minikube

# Remove the installation package
rm minikube-linux-amd64
```

### **Start the Kubernetes cluster**

When `minikube start` runs, Minikube uses Docker to create and configure the Kubernetes control plane, including the API server, scheduler, controller manager, and kubelet services.

Start the cluster:

```shell
minikube start
```

### **Verify the cluster**

Check the cluster status:

```shell
minikube status
```

The command displays output similar to the following example:

```txt
host: Running
kubelet: Running
apiserver: Running
```

To verify that the Minikube node is available, run:

```shell
kubectl get nodes
```

The command displays output similar to the following example:

```txt
NAME       STATUS   ROLES           AGE
minikube   Ready    control-plane   ...
```

A `Ready` status indicates that the cluster is running and ready to host workloads.

<Info>
  #### **Minikube in the deployment architecture**

  The infrastructure now includes the following layers:

  - Infrastructure: Google Cloud VM
  - Containerization: Docker Engine
  - Kubernetes distribution: Minikube
  - Cluster resources: pods, services, and deployments
</Info>

The next step installs `kubectl`, the command-line tool used to interact with and manage Kubernetes resources.

## Step 5: **Install** `kubectl `**and verify cluster connectivity**

`kubectl` is the command-line tool for Kubernetes. It communicates with the Kubernetes API server and enables the following operations:

- Creating deployments
- Managing pods
- Exposing services
- Viewing logs
- Monitoring cluster resources
- Creating namespaces
- Working with Kubernetes secrets

### **Install** `kubectl`

Install `kubectl` by using Snap:

```shell
sudo snap install kubectl --classic
```

To install `kubectl` by using the official Kubernetes binary instead, see the [Kubernetes documentation](https://kubernetes.io/docs/tasks/tools/).

### **Verify the installation**

Verify that `kubectl` is available:

```shell
kubectl version --client
```

The command displays output similar to the following example:

```txt
Client Version: vX.X.X
```

This confirms that `kubectl` is installed.

### **Verify cluster connectivity**

Verify that `kubectl` can communicate with the Minikube cluster.

Check the cluster status:

```shell
minikube status
```

The command displays output similar to the following example:

```txt
host: Running
kubelet: Running
apiserver: Running
```

View the nodes managed by Kubernetes:

```shell
kubectl get nodes
```

The command displays output similar to the following example:

```txt
NAME       STATUS   ROLES           AGE
minikube   Ready    control-plane   ...
```

A `Ready` status indicates that the cluster is running and ready to host workloads.

View cluster information:

```shell
kubectl cluster-info
```

The command displays details about the Kubernetes control plane and confirms that the API server is accessible.

### **Verify Minikube and Docker integration**

Minikube uses Docker as its container runtime. To confirm that the cluster components are running, check the active Docker containers:

```shell
docker ps
```

The output lists containers associated with Minikube and Kubernetes, indicating that the cluster is running correctly.

<Info>
  #### `kubectl `**in the deployment architecture**

  The deployment architecture now consists of the following layers:

  - Infrastructure: Google Cloud VM
  - Containerization: Docker Engine
  - Kubernetes distribution: Minikube
  - Cluster management: `kubectl`
  - API endpoint: Kubernetes API server
</Info>

The next step connects the VM to the GitHub repository to pull the application source code for deployment.

# Start the Minikube cluster

minikube start

\`\`\`

At this point you have a running single-node Kubernetes cluster on your VM.

## Step 4: Clone your code and build the image

Next, pull your application code onto the VM and build the Docker image that Kubernetes will run.

\`\`\`bash

# Clone your repository

git clone [https://github.com/YOUR\_USERNAME/YOUR\_REPO\_NAME.git](https://github.com/YOUR_USERNAME/YOUR_REPO_NAME.git)

cd YOUR\_REPO\_NAME

# Point Docker CLI to Minikube's Docker daemon

eval \$(minikube docker-env)

# Build the Docker image for your app

docker build -t llmops-app:latest .

\`\`\`

Pointing Docker to the Minikube daemon ensures the image is available directly inside the cluster without needing an external registry.

---

## Step 5: Deploy the application to Kubernetes

With the image built, you can deploy the Streamlit app as a Kubernetes deployment and service.

First, create a Kubernetes Secret to store API keys and tokens:

\`\`\`bash

kubectl create secret generic llmops-secrets \\

--from-literal=GROQ\_API\_KEY="your-groq-key-here" \\

--from-literal=HUGGINGFACEHUB\_API\_TOKEN="your-hf-token-here"

\`\`\`

Then apply your Kubernetes manifest:

\`\`\`bash

# Apply deployment and service manifests

kubectl apply -f llmops-k8s.yaml

# Verify that the pod is running

kubectl get pods

\`\`\`

Wait until the pod status shows `Running` before proceeding.

---

## Step 6: Expose the app to the internet

Streamlit runs on port `8501` inside the cluster. You now need to make it reachable from the public internet.

Because this setup uses Minikube instead of a cloud-managed Kubernetes service, you will use a combination of `minikube tunnel` and `kubectl port-forward`.

Open two SSH sessions to the VM:

### Terminal 1

\`\`\`bash

# Create a network tunnel from the VM into Minikube

minikube tunnel

\`\`\`

Leave this command running; it establishes routes for LoadBalancer-type services.

### Terminal 2

\`\`\`bash

# Forward the Kubernetes service port to an external port on the VM

kubectl port-forward svc/llmops-service 8501:80 --address 0.0.0.0

\`\`\`

This forwards traffic from port `8501` on the VM to port `80` on the Kubernetes service.

Finally, in the Google Cloud Console, note the **External IP** of your VM and open it in your browser:

\`\`\`text

http://\<YOUR\_EXTERNAL\_IP\>:8501

\`\`\`

You should now see your live Streamlit application running on Google Cloud.

## Step 7: Configure Grafana Cloud monitoring (optional)

To monitor resource usage and the health of your cluster, you can integrate Grafana Cloud.

1. Go to [https://grafana.com/](https://grafana.com/) and create a Grafana Cloud account.
2. In Grafana, open **Observability → Kubernetes → Start sending data**.
3. Start a new integration, giving it a name such as `minikube` and `monitoring`.
4. Generate an access token (for example, `minikube-token`) and store it securely.
5. Grafana will provide a `helm upgrade --install ...` command that deploys the required agents.

On your VM:

\`\`\`bash

# Create a dedicated namespace for monitoring components

kubectl create ns monitoring

\`\`\`

Then paste and run the `helm upgrade` command provided by Grafana. After a few minutes, verify that the agents are running:

\`\`\`bash

kubectl get pods -n monitoring

\`\`\`

Back in Grafana Cloud, open the Kubernetes dashboards to view CPU, memory, and network metrics for your cluster and application.

## Step 8: Updating the deployed application

When you push a bug fix or feature update to GitHub, you can roll out the change with a short sequence of commands on the VM.

From your application directory:

\`\`\`bash

# 1. Pull the latest code

git pull origin main

# 2. Ensure Docker is pointing to Minikube's daemon (if you opened a new session)

eval \$(minikube docker-env)

# 3. Rebuild the Docker image with the updated code

docker build -t llmops-app:latest .

# 4. Restart the Kubernetes deployment to use the new image

kubectl rollout restart deployment llmops-app

\`\`\`

Kubernetes will roll out updated pods, and once they are ready, refreshing your browser will show the new version of your Streamlit app.
