If you have built a Streamlit-based Machine Learning or AI application and are wondering how to make it accessible on the internet, this guide walks through a complete cloud-native deployment workflow. By the end, you will deploy a Streamlit app to a Google Cloud VM, run it inside a Minikube Kubernetes cluster using Docker, expose it to the public internet, and optionally monitor it with Grafana Cloud.

## Architecture overview

Before writing any commands, it helps to understand the high-level architecture of the system you are about to deploy.

![Gcp K8s Architecture](/images/gcp_k8s_architecture.png)

The main components are:

<Steps>
  <Step title="Google Cloud VM instance">
    An Ubuntu-based virtual machine provisioned on Google Cloud Compute Engine. This VM acts as the infrastructure layer. It hosts Docker, Minikube, and the Kubernetes workloads that run the Streamlit application.
  </Step>
  <Step title="Minikube Kubernetes cluster">
    A single-node Kubernetes cluster running on the VM that manages pods, services, and deployments for the application.
  </Step>
  <Step title="Docker">
    A containerization platform that bundles the application source code, Python dependencies, and runtime configuration into a portable container image that runs consistently across environments.
  </Step>
  <Step title="Streamlit app">
    A Python web application served on port 8501 inside a Kubernetes pod.
  </Step>
  <Step title="Grafana Cloud">
    A managed observability platform for monitoring Kubernetes resources in real time. Monitored resources include nodes, pods, deployments, CPU usage, memory utilization, and overall cluster health.
  </Step>
</Steps>

## Step 1: Prepare the repository

Before using Google Cloud, verify that the code is organized and pushed to GitHub.

The GitHub repository should include:

- Application code, for example`app.py`, `src/`.
- A `requirements.txt`file listing all Python dependencies with compatible versions.
- A `Dockerfile`describing how to build the container image for the application.
- A Kubernetes manifest file for the deployment and service, for example`llmops-k8s.yaml`.

Keeping these assets version-controlled makes it easy to reproduce deployments and roll back safely.

## Step 2: Create a Google Cloud VM

Create the virtual machine that will host the Kubernetes cluster and application.

1. Sign in to the Google Cloud Console.
2. Click **Compute Engine \> VM instances**, and then click **Create instance**.
3. Set a descriptive name, for example ,`streamlit-app-server`.
4. Under **Machine configuration**, select an E2 machine type.<br />If you are running ML workloads, ensure that the machine type has at least 16 GB of RAM, for example, `e2-standard-4`.
5. Under **Boot disk**, click **Change**, select Ubuntu 24.04 LTS, and increase the disk size to approximately 256 GB to accommodate Docker images and model files.
6. In the **Firewall** section, enable **Allow HTTP traffic** and **Allow HTTPS traffic**.
7. Click **Create** and wait until the VM is ready.
8. Click the **SSH** button next to the instance to open a terminal in the browser.

<Info>
  #### What happens during VM provisioning?

  At this stage, Google Cloud provisions a dedicated Ubuntu server for the deployment. This server hosts the following components:

  - Docker Engine
  - Minikube Kubernetes cluster
  - `kubectl` command-line tool
  - Streamlit application containers

  The VM supports the entire deployment architecture.
</Info>

## Step 3: **Install and configure Docker**

Docker is the containerization platform that packages the Streamlit application, its dependencies, and runtime environment into a portable image that can be deployed consistently across environments.

Docker is also a prerequisite for Minikube, because Minikube uses Docker as its container runtime when creating the Kubernetes cluster.

### **Install Docker**

Complete the following substeps to install Docker Engine.

#### **Install Docker dependencies**

Update the package repository and install the utilities required to download Docker packages securely:

```shell
sudo apt-get update
sudo apt-get install -y ca-certificates curl
```

#### **Add the official Docker GPG key**

Ubuntu verifies package authenticity using GPG keys. The official Docker GPG key ensures that the packages being installed come from a trusted source:

```shell
sudo install -m 0755 -d /etc/apt/keyrings
sudo curl -fsSL https://download.docker.com/linux/ubuntu/gpg \
  -o /etc/apt/keyrings/docker.asc
sudo chmod a+r /etc/apt/keyrings/docker.asc
```

#### **Add the Docker package repository**

Add the official Docker package repository to the system so that Ubuntu can download the latest Docker packages directly from the Docker-maintained repository:

```shell
echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.asc] \
  https://download.docker.com/linux/ubuntu \
  $(. /etc/os-release && echo "$VERSION_CODENAME") stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

sudo apt-get update
```

#### **Install Docker Engine**

Install Docker Engine and its associated plugins:

```shell
sudo apt-get install -y docker-ce docker-ce-cli \
  containerd.io docker-buildx-plugin docker-compose-plugin
```

The installation includes the following components:

- Docker Engine
- Docker CLI
- Container runtime (containerd)
- Buildx plugin
- Docker Compose plugin

#### **Allow Docker usage without** `sudo`

By default, Docker commands require elevated privileges. Add the current user to the Docker group to run Docker commands without using `sudo`:

```shell
sudo usermod -aG docker $USER
```

> **Note:** Run `newgrp docker`, or log out and log in again, for the group membership change to take effect.

#### **Enable Docker on system startup**

Configure Docker services to start automatically when the VM reboots:

```shell
sudo systemctl enable docker.service
sudo systemctl enable containerd.service
```

#### **Verify the installation**

To verify that Docker is installed correctly, run:

```text
docker run hello-world
```

<Note>
  **Note:** If Docker is run without `sudo`, a permission-denied error might occur. This happens because the current user has not yet been added to the Docker user group. The Docker post-installation steps resolve this issue by granting the appropriate permissions.
</Note>

After the configuration is complete, the command displays:

```text
Hello from Docker!
```

This verifies that the Docker daemon is running and can pull and run containers.

#### **Validate the Docker configuration**

To confirm that Docker is operating correctly, run:

```shell
systemctl status docker
docker ps
docker ps -a
```

Expected results:

- The Docker service status shows `active (running)`.
- `docker ps` returns no running containers.
- `docker ps -a` displays the previously run `hello-world` container in an exited state.

<Info>
  #### **Docker in the deployment architecture**

  At this stage, the deployment architecture consists of the following layers:

  - **Google Cloud VM** — the infrastructure layer
  - **Docker Engine** — installed and running on the VM
  - **Container runtime** — ready to build images and run containers

  The VM can now build Docker images and run containers. In the next step, Minikube uses Docker to create a Kubernetes cluster that hosts the Streamlit application.
</Info>

## Step 4: **Install Minikube and create a Kubernetes cluster**

Minikube is a single-node Kubernetes distribution that creates a Kubernetes cluster inside the Google Cloud VM. It uses Docker as its container runtime, making Docker a prerequisite for this step.

### **Kubernetes orchestration**

Rather than running containers directly on the VM, Kubernetes provides an orchestration layer that manages the following resources:

- pods
- deployments
- services
- secrets
- namespaces

Kubernetes enables applications to be deployed, updated, and monitored through a consistent, declarative management model.

### **Install Minikube**

Download and install the Minikube binary:

```shell
# Download Minikube
curl -LO https://storage.googleapis.com/minikube/releases/latest/minikube-linux-amd64

# Install Minikube
sudo install minikube-linux-amd64 /usr/local/bin/minikube

# Remove the installation package
rm minikube-linux-amd64
```

### **Start the Kubernetes cluster**

When `minikube start` runs, Minikube uses Docker to create and configure the Kubernetes control plane, including the API server, scheduler, controller manager, and kubelet services.

Start the cluster:

```shell
minikube start
```

### **Verify the cluster**

Check the cluster status:

```shell
minikube status
```

The command displays output similar to the following example:

```txt
host: Running
kubelet: Running
apiserver: Running
```

To verify that the Minikube node is available, run:

```shell
kubectl get nodes
```

The command displays output similar to the following example:

```txt
NAME       STATUS   ROLES           AGE
minikube   Ready    control-plane   ...
```

A `Ready` status indicates that the cluster is running and ready to host workloads.

## Step 6: **Connect the VM to GitHub and clone the repository**

This step brings the application source code onto the VM by cloning the GitHub repository. The VM can then pull updated code when changes are pushed from the local development machine.

### **Clone the repository**

Clone the repository onto the VM:

```shell
git clone https://github.com/<username>/<repository>.git
```

List the directory contents to confirm the clone succeeded:

```shell
ls
```

Change into the project directory:

```shell
cd <repository>
ls
```

The directory contains files similar to the following example:

```txt
Dockerfile
requirements.txt
app.py
src/
llmops-k8s.yaml
```

This confirms that the application source code is available on the VM.

### **Configure Git identity**

Configure Git with the GitHub username and email address:

```shell
git config --global user.email "username@example.com"
git config --global user.name "<github-username>"
```

This configuration associates commits made from the VM with the correct GitHub account.

### **Authenticate by using a GitHub personal access token**

When pushing changes from the VM, GitHub requires authentication by using a personal access token (PAT) instead of an account password.

To generate a token, navigate to:

**GitHub \> Settings \> Developer settings \> Personal access tokens**

Grant read and write access to the repository, and store the token in a protected location. Git uses this token when it prompts for authentication credentials.

### **Synchronize changes between environments**

To confirm that the VM and the GitHub repository are connected, pull the latest changes:

```shell
git pull origin main
```

When code is updated on the local machine, push the changes to GitHub:

```shell
git add .
git commit -m "Updated application"
git push origin main
```

To retrieve those updates on the VM, run:

```shell
git pull origin main
```

## Step 7: **Configure Google Cloud firewall rules**

By default, Google Cloud blocks all incoming network traffic. To make the Streamlit application accessible from the internet, configure a firewall rule that allows external traffic to reach the VM.

### **Firewall rule requirements**

The Streamlit application is exposed through a Kubernetes service and a Minikube tunnel. The firewall rule must allow inbound connections on the ports used by those services.

#### **Create a firewall rule**

In the Google Cloud Console, navigate to:

**VPC network \> Firewall \> Create firewall rule**

Configure the rule with the following settings:

| Setting | Value |
| :-- | :-- |
| Direction | Ingress |
| Action | Allow |
| Targets | All instances in the network |
| Source filter | IPv4 ranges |
| Source IPv4 range | `0.0.0.0/0` |

The source IPv4 range `0.0.0.0/0` allows inbound connections from any IP address.

### **Allow application traffic**

Configure the firewall rule to allow traffic on the following ports:

- `80` (HTTP)
- `443` (HTTPS)
- `8501` (Streamlit application)

These ports allow requests to reach the Kubernetes service that exposes the application.

### **Firewall rule effect**

Before the firewall rule is applied, all incoming traffic is blocked at the network boundary and does not reach the VM.

After the firewall rule is applied, the traffic flow is:

- Internet
- Google Cloud firewall (rule allows ingress)
- Google Cloud VM
- Kubernetes cluster

External users can reach services running inside the VM, making it possible to expose the Streamlit application publicly.

### **Verify the firewall rule**

After creating the rule, verify that the firewall rule appears in the **Firewall rules** list and that its status is enabled.

When the Kubernetes LoadBalancer service is created in a later step, traffic can reach the application from outside the Google Cloud network.

## Step 8: B**uild the Docker image and deploy the application to Kubernetes**

This step builds the Docker image from the application source code and deploys it into the Minikube Kubernetes cluster.

### **Point Docker to the Minikube environment**

Before building the image, configure Docker to use the Minikube internal Docker daemon. Images built in this context are available directly to Kubernetes without requiring a push to an external container registry.

Run the following command:

```shell
eval $(minikube docker-env)
```

### **Build the Docker image**

Change into the project root directory, then build the image:

```shell
docker build -t llmops-app:latest .
```

During the build, Docker performs the following operations:

1. Pulls the base Python image.
2. Creates a working directory inside the container.
3. Copies application files into the image.
4. Installs dependencies from `requirements.txt`.
5. Exposes the Streamlit port (`8501`).
6. Configures the container startup command.

The resulting image is tagged as `llmops-app:latest` and contains everything required to run the Streamlit application.

### **Verify the Docker image**

Verify that the image was created successfully:

```shell
docker images
```

The command displays output similar to the following example:

```txt
REPOSITORY    TAG      IMAGE ID
llmops-app    latest   xxxxxxxxx
```

When the image appears in the list, the build is complete.

### **Create Kubernetes secrets**

Storing credentials in source code or repositories exposes them to unauthorized access. Use Kubernetes secrets to inject API keys as environment variables into the application pods. The application reads the credentials from the environment variables, keeping them out of the source code.

Create the secret:

```shell
kubectl create secret generic llmops-secrets \
  --from-literal=GROQ_API_KEY="<GROQ_API_KEY>" \
  --from-literal=HUGGINGFACEHUB_API_TOKEN="<HUGGINGFACE_TOKEN>"
```

The deployment manifest references these secrets and injects them as environment variables inside the container at runtime.

### **Deploy the application**

Apply the Kubernetes deployment manifest:

```shell
kubectl apply -f llmops-k8s.yaml
```

The command displays output similar to the following example:

```txt
deployment.apps/llmops created
service/llmops-service created
```

The `llmops-k8s.yaml` file defines two Kubernetes resources:

- **Deployment** — creates and manages the application pods
- **Service** — exposes the pods internally and externally

Kubernetes schedules the pod and starts the Streamlit container by using the Docker image built in the previous step.

### **Verify pod status**

Verify that the application pod is running:

```shell
kubectl get pods
```

The command displays output similar to the following example:

```txt
NAME                          READY   STATUS
llmops-deployment-xxxxx       1/1     Running
```

A status of `Running` indicates that the application started successfully inside Kubernetes.

### **Inspect Kubernetes resources**

Run the following commands to verify that Kubernetes created all resources defined in the deployment manifest.

Check deployments:

```shell
kubectl get deployments
```

Check services:

```shell
kubectl get services
```

### **Deployment architecture**

The deployment architecture now consists of the following layers:

- Infrastructure: Google Cloud VM
- Kubernetes distribution: Minikube
- Resource controller: Kubernetes deployment
- Workload unit: pod
- Runtime: Docker container
- Application: Streamlit

The application is running inside Kubernetes but is not yet accessible from the internet. The next step exposes the service by using Minikube Tunnel and port forwarding, making the Streamlit application accessible through the public IP address of the VM.

## Step 9: **Expose the application and verify access**

The application is running inside a Kubernetes pod but is not yet accessible outside the cluster. Complete the following tasks to expose the service and verify external access:

- Create a network route from Minikube to the external network.
- Expose the Kubernetes service.
- Forward requests to the Streamlit application on port `8501`.

### **Verify the service**

The deployment manifest defines a LoadBalancer service, which is required for external access through a public IP address. Confirm that Kubernetes created the service:

```shell
kubectl get services
```

The command displays output similar to the following example:

```txt
NAME              TYPE           CLUSTER-IP
llmops-service    LoadBalancer   10.x.x.x
```

### **Start the Minikube tunnel**

In a cloud-managed Kubernetes environment, a LoadBalancer service automatically receives an external IP address. In Minikube, the `minikube tunnel` command simulates this functionality by creating external networking routes and assigning external IP addresses to LoadBalancer services.

Open a second terminal session and run:

```shell
minikube tunnel
```

This terminal session must remain open while the application is being accessed.

### **Forward the service port**

Open a third terminal session and run:

```shell
kubectl port-forward svc/llmops-service 8501:80 --address 0.0.0.0
```

This command forwards traffic as follows:

- External port `8501` on the VM
- Kubernetes service port `80`
- Streamlit container

The `--address 0.0.0.0` flag allows traffic from external sources. Without this flag, access is limited to localhost.

### **Retrieve the external IP address of the VM**

In the Google Cloud Console, navigate to:

**Compute Engine \> VM instances**

Copy the external IP address of the VM. The address has the following format:

```txt
34.xxx.xxx.xxx
```

### **Access the application**

In a browser, go to:

```txt
http://<external-ip>:8501
```

For example:

```txt
http://34.xxx.xxx.xxx:8501
```

When the page loads for the first time, the application might take a short time to become ready because the Streamlit application is starting, the recommendation pipeline is loading, and the models and dependencies are being initialized.

### **Test the deployment**

After the application loads, submit a sample query to verify that it is working correctly. For example, enter a query such as:

```txt
Recommend me a AI system similar to RTX
```

A successful response confirms the following:

- The Docker image was built correctly.
- The Kubernetes deployment completed successfully.
- Kubernetes secrets are available to the application.
- The port forwarding is active.
- External access is configured.

### **Verify running resources**

Check the application pod and service:

```shell
kubectl get pods
kubectl get services
```

The commands display output similar to the following example:

```txt
NAME                        READY   STATUS
llmops-deployment-xxxxx     1/1     Running
```

A status of `Running` indicates that Kubernetes is managing the Streamlit application successfully.

## Step 10: Configure Grafana Cloud monitoring (optional)

To monitor resource usage and the health of your cluster, you can integrate Grafana Cloud.

1. Go to [https://grafana.com/](https://grafana.com/) and create a Grafana Cloud account.
2. In Grafana, open **Observability → Kubernetes → Start sending data**.
3. Start a new integration, giving it a name such as `minikube` and `monitoring`.
4. Generate an access token (for example, `minikube-token`) and store it securely.
5. Grafana will provide a `helm upgrade --install ...` command that deploys the required agents.
